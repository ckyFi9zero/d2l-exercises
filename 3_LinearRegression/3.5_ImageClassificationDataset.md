### 1. 减少batch_size（如减少到1）是否会影响读取性能？

**结论：** 会显著降低读取性能。

**分析：**

* **I/O 开销：** 当 `batch_size=1` 时，数据迭代器每次只能从磁盘读取一张图片。由于现代操作系统和硬件（如 SSD）对大块数据的吞吐量做了优化，频繁的小块读取会产生极大的 I/O 等待开销。
* **Python 循环开销：** 在代码实现中，`DataLoader` 每次迭代都会产生一定的 Python 调用开销。如果 `batch_size` 过小，Python 解释器在处理循环逻辑（如进程间通信、数据拼装）上花费的时间甚至会超过模型计算的时间。
* **硬件利用率：** 现代 CPU 和 GPU 的设计初衷是并行处理。`batch_size=1` 无法利用单指令多数据流（SIMD）或 GPU 的并行计算能力，导致硬件资源处于闲置状态。

### 2. 数据迭代器的性能非常重要。当前的实现足够快吗？探索各种选择来改进它。

在代码中，我们定义了 `get_dataloader_workers()` 来开启多进程读取。当前的实现在数据量较小时（如 Fashion-MNIST）是足够的，但在处理更大规模的数据或进行复杂预处理时，仍有改进空间：

* **当前实现（基础版）：**
  代码中使用了 `num_workers=4`。这意味着有 4 个子进程在后台并行预取数据，这比单进程要快得多。
* **进阶改进方案：**

  1. **`pin_memory=True`：** 在 `DataLoader` 中开启此选项，可以将数据直接加载到页锁定内存中，这样数据转移到 GPU 的速度会更快。
  2. **调整 `num_workers`：** 通常将其设置为 CPU 核心数。如果设置得太小，CPU 处理速度慢；如果设置得太大，进程间切换的开销反而会拖累系统。
  3. **`prefetch_factor`：** 调整每个 worker 提前加载的 batch 数量，平衡内存占用和读取速度。
  4. **`persistent_workers=True`：** 在一个 Epoch 结束后不销毁 worker 进程，减少下一轮迭代初始化进程的时间开销。
  5. **存储媒介优化：** 确保数据集存储在高速 NVMe SSD 上，或者在内存充足的情况下，将数据集放入虚拟磁盘（RAMDisk）。

### 3. 查阅框架的在线API文档。还有哪些其他数据集可用？

除了代码中使用的 `torchvision.datasets.FashionMNIST`，`torchvision` 框架还提供了极其丰富的数据集 API。查阅文档后，常见的可用数据集包括：

* **经典分类数据集：**

  * `MNIST`：原始手写数字集。
  * `CIFAR10` / `CIFAR100`：更具挑战性的 32x32 彩色图像分类。
  * `ImageNet`：深度学习最著名的超大规模图像分类数据集。
* **目标检测与分割：**

  * `COCO` (Common Objects in Context)：用于检测、分割和字幕生成的标准数据集。
  * `VOC` (Pascal Visual Object Classes)：经典的目标检测数据集。
* **人脸识别：**

  * `CelebA`：包含大量人脸属性标注的数据集。
* **场景理解：**

  * `Places365`：用于场景识别的大规模数据集。
